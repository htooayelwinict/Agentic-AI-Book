# ၀၄။ အမှားအယွင်းများကြားမှ လျှောက်လှမ်းခြင်းနှင့် ကိုယ်စားလှယ်၏ ကွင်းဆက်

### အဆုံးမရှိသော သံသရာနှင့် ဆင်ခြင်တုံတရား

ကျွန်ုပ်တို့သည် ယခင်အခန်းများတွင် Agent တစ်ခု၏ "ဦးနှောက်" နှင့် "လက်" များအကြောင်းကို အကျယ်တဝင့် ဆွေးနွေးခဲ့ကြသည်။ သို့သော် ထိုအစိတ်အပိုင်းများကို စုစည်းလိုက်ရုံဖြင့် အသိဉာဏ်ရှိသော ကိုယ်စားလှယ်တစ်ခု ဖြစ်မလာသေးချေ။ အမှန်တကယ် အလုပ်လုပ်နိုင်သော Agent တစ်ခု ဖြစ်လာရန်အတွက် **Agent Loop** ဟု ခေါ်သော "စဉ်းစားဆင်ခြင်မှု ကွင်းဆက်" တစ်ခု လိုအပ်သည်။ ဤသည်မှာ လူသားတို့ အလုပ်တစ်ခုကို လုပ်ဆောင်ရာတွင် "စဉ်းစားသည်၊ လုပ်သည်၊ လွဲချော်သည်၊ ပြန်ပြင်သည်" ဟူသော အဆင့်ဆင့်သော ဖြစ်စဉ်နှင့် အလွန်ပင် တူညီလှသည်။

ရိုးရိုး Software များသည် မျဉ်းဖြောင့်အတိုင်း သွားကြသည်။ ၎င်းတို့သည် အမှားနှင့် တွေ့ပါက ရပ်တန့်သွားတတ်ကြသည်။ သို့သော် Agentic AI ၏ ထူးခြားချက်မှာ အမှားကို တွေ့ရှိသောအခါ "အဘယ်ကြောင့် မှားသနည်း" ဟု ပြန်လည် ဆင်ခြင်နိုင်ခြင်း (Self-Reflection) နှင့် ထိုအမှားကို ကျော်လွှားရန် နည်းလမ်းသစ်ကို ရှာဖွေနိုင်ခြင်း ဖြစ်သည်။ ဤသည်ကိုပင် **Reasoning Loop** ဟု ခေါ်ဆိုခြင်း ဖြစ်သည်။ ဤကွင်းဆက်သည် Agent အား "စာရေးတော်" အဆင့်မှ "ပြဿနာ ဖြေရှင်းသူ" အဆင့်သို့ မြှင့်တင်ပေးလိုက်ခြင်းပင် ဖြစ်ပေတော့သည်။

### ReAct: စဉ်းစားခြင်းနှင့် လုပ်ဆောင်ခြင်း၏ ပေါင်းစပ်မှု

Agentic AI လောကတွင် အကျော်ကြားဆုံးသော နည်းလမ်းတစ်ခုမှာ **ReAct (Reason + Act)** ပုံစံ ဖြစ်သည်။ ဤနည်းလမ်းသည် Agent အား အလုပ်တစ်ခုကို မလုပ်မီ "ငါ ဘာလုပ်တော့မည်" ဟု အရင် စဉ်းစားခိုင်းခြင်း (Thought)၊ ထို့နောက် လက်တွေ့ လုပ်ဆောင်ခိုင်းခြင်း (Action) နှင့် ရရှိလာသော ရလဒ်ကို ကြည့်ရှုခိုင်းခြင်း (Observation) တို့ကို အပြန်အလှန် လုပ်ဆောင်စေသည်။

ဥပမာအားဖြင့်၊ သင်က Agent ကို "မြန်မာနိုင်ငံ၏ လက်ရှိ လူဦးရေကို ရှာပေးပါ" ဟု ခိုင်းလိုက်သည် ဆိုပါစို့။
၁။ **Thought (စဉ်းစားချက်):** "ငါ့ရဲ့ မှတ်ဉာဏ်ထဲမှာ လက်ရှိ လူဦးရေ အတိအကျ မရှိဘူး။ ဒါကြောင့် Google မှာ ရှာဖို့ လိုအပ်တယ်။"
၂။ **Action (လုပ်ဆောင်ချက်):** `google_search("Myanmar population 2024")` ဟူသော Tool ကို ခေါ်ယူသည်။
၃။ **Observation (စောင့်ကြည့်ချက်):** ရှာဖွေမှု ရလဒ်တွင် "၅၅ သန်းကျော်" ဟု တွေ့ရှိရသည်။
၄။ **Thought (စဉ်းစားချက်):** "အချက်အလက် ရပြီ။ ဒါပေမဲ့ ဒါဟာ တရားဝင် သန်းခေါင်စာရင်းလားဆိုတာ ထပ်စစ်သင့်တယ်။"

ဤသို့ဖြင့် Agent သည် မိမိ၏ အလုပ်ကို မိမိဘာသာ ပြန်လည် ဝေဖန်ဆန်းစစ်ရင်း ပန်းတိုင်သို့ ရောက်အောင် သွားခြင်း ဖြစ်သည်။ ဤသည်မှာ "လူမိုက်" တစ်ယောက်ကို အလုပ်ခိုင်းရာတွင် "မင်း ဘာလုပ်နေလဲဆိုတာ ငါ့ကို အမြဲပြောပြပါ" ဟု စည်းကမ်းထုတ်ထားခြင်းနှင့် တူသည်။ ဤသို့ ထုတ်ဖော် ပြောဆိုခိုင်းခြင်း (Chain of Thought) သည် LLM ၏ ယောင်မှားမှု (Hallucination) ကို သိသိသာသာ လျှော့ချပေးနိုင်ကြောင်း ကျွန်ုပ်တို့ တွေ့ရှိရသည်။

### အမှားကို အခွင့်အလမ်းအဖြစ် ပြောင်းလဲခြင်း

Agent တစ်ခု၏ စွမ်းဆောင်ရည်ကို တိုင်းတာရာတွင် ၎င်းသည် အလုပ်ကို မည်မျှ မြန်မြန် ပြီးသနည်း ဆိုသည်ထက်၊ အမှားနှင့် တွေ့သောအခါ မည်သို့ တုံ့ပြန်သနည်း ဆိုသည်က ပို၍ အရေးကြီးသည်။ ဥပမာ - Agent က Tool တစ်ခုကို ခေါ်ယူလိုက်သော်လည်း ထို Tool က Error တက်သွားသည် ဆိုပါစို့။ ရိုးရိုး Software ဆိုလျှင် "Error: 404" ဟု ပြပြီး ရပ်သွားပေလိမ့်မည်။

သို့သော် Agentic Loop ထဲတွင်မူ ထို Error သည် Agent အတွက် "အချက်အလက်သစ်" (Observation) တစ်ခု ဖြစ်လာသည်။ Agent က "ဪ... ဒီနည်းလမ်းကတော့ မအောင်မြင်ဘူးပဲ။ တခြားနည်းလမ်းနဲ့ ကြိုးစားကြည့်ရမယ်" ဟု ဆုံးဖြတ်နိုင်သည်။ ဤသည်ကို **Error Recovery** ဟု ခေါ်သည်။ ဤစွမ်းရည်သည် Agent ကို အလွန်ပင် အစွမ်းထက်စေသည်။ ၎င်းသည် လူသားတို့၏ ကြီးကြပ်မှု မပါဘဲ နာရီပေါင်းများစွာ၊ ရက်ပေါင်းများစွာ အလုပ်လုပ်နိုင်စေရန် ဖန်တီးပေးထားသော "ဇွဲလုံ့လ" ပင် ဖြစ်သည်။

သို့သော် ဤ "ဇွဲ" သည် တစ်ခါတစ်ရံတွင် အန္တရာယ် ရှိတတ်သည်။ အကယ်၍ Agent သည် မဖြေရှင်းနိုင်သော ပြဿနာတစ်ခုကို အဖန်တလဲလဲ ကြိုးစားနေပါက ၎င်းသည် **Infinite Loop** (အဆုံးမရှိသော သံသရာ) ထဲသို့ ရောက်သွားပြီး သင်၏ API ကုန်ကျစရိတ်များကို မြှင့်တင်ပေးပေလိမ့်မည်။ ထို့ကြောင့် အင်ဂျင်နီယာတစ်ယောက်အနေဖြင့် ကျွန်ုပ်တို့သည် "အကြိမ်ရေ ကန့်သတ်ချက်" (Max Iterations) များကို မဖြစ်မနေ ထည့်သွင်းပေးရမည်။ "မင်း ၁၀ ကြိမ်အထိ ကြိုးစားလို့ မရရင်တော့ လက်လျှော့လိုက်တော့" ဟူသော အမိန့်မျိုး ဖြစ်သည်။

### စဉ်းစားဆင်ခြင်မှု၏ ကုန်ကျစရိတ်

Agentic Loop တစ်ခုကို ပတ်တိုင်း ကျွန်ုပ်တို့သည် LLM ထံသို့ Token များ ပေးပို့နေရသည်။ ဆိုလိုသည်မှာ အလုပ်တစ်ခုကို ပြီးမြောက်ရန်အတွက် Agent က အကြိမ်ကြိမ် စဉ်းစားလေလေ၊ သင်၏ ပိုက်ဆံ ကုန်လေလေ ဖြစ်သည်။ ဤသည်မှာ "အချိန်နှင့် ငွေကြေး" အားပြိုင်မှုပင် ဖြစ်သည်။ ကျွန်ုပ်တို့သည် အလွန်တော်သော Agent ကို လိုချင်သော်လည်း၊ ထို Agent က ရိုးရိုးလေး ဖြေရမည့် မေးခွန်းကိုပင် ၁၅ မိနစ်ခန့် စဉ်းစားနေပါက လက်တွေ့တွင် အသုံးမဝင်လှချေ။

ထို့ကြောင့် ကျွန်ုပ်တို့သည် **Efficiency** (ထိရောက်မှု) ကိုလည်း ထည့်သွင်း စဉ်းစားရမည်။ Agent အား အရာရာကို အသေးစိတ် စဉ်းစားခိုင်းမည့်အစား၊ အရေးကြီးသော အချက်များတွင်သာ Loop ပတ်စေခြင်း၊ သို့မဟုတ် သေးငယ်ပြီး မြန်ဆန်သော Model (Small Language Models) များကို အချို့သော အဆင့်များတွင် အစားထိုး အသုံးပြုခြင်း စသည့် နည်းဗျူဟာများကို သုံးစွဲရမည်။ ဤသည်မှာ "စာရေးတော်" ကို ခိုင်းစေရာတွင် အလုပ်၏ အတိမ်အနက်အလိုက် လူမှန် နေရာမှန် ခွဲဝေပေးခြင်းနှင့် တူပေသည်။

### နိဂုံး - ကွင်းဆက်၏ အဆုံးသတ်

Agentic Loop သည် Agentic AI ၏ နှလုံးခုန်သံ ဖြစ်သည်။ ၎င်းသည် စက်ပစ္စည်းများကို "စဉ်းစားတတ်သူ" များအဖြစ် ပြောင်းလဲပေးလိုက်ခြင်း ဖြစ်သည်။ သို့သော် ထိုနှလုံးခုန်သံသည် စည်းချက်ကျရန် လိုအပ်သည်။ အလွန်အကျွံ ခုန်နေပါက စနစ်ကြီးတစ်ခုလုံး ပူလောင်သွားနိုင်သလို၊ နှေးကွေးနေပါကလည်း အလုပ်မတွင်နိုင်ချေ။

နောက်အခန်းတွင်မူ ဤ Loop များကို စနစ်တကျ ထိန်းကျောင်းပေးမည့်၊ Agent ၏ မှတ်ဉာဏ်နှင့် ပတ်ဝန်းကျင်ကို စီမံပေးမည့် **Agent Harness** (ထိန်းကျောင်းမှု စနစ်) အကြောင်းကို ဆက်လက် ဆွေးနွေးသွားပါမည်။ ဤသည်မှာ ကျွန်ုပ်တို့၏ "စာရေးတော်" အတွက် လုံခြုံပြီး စွမ်းဆောင်ရည် မြင့်မားသော "ရုံးခန်း" တစ်ခု တည်ဆောက်ပေးခြင်းပင် ဖြစ်ပေတော့သည်။

---
**စာဖတ်သူအတွက် Source Notes:**
- **ReAct Pattern:** A prompting technique that combines reasoning traces and task-specific actions in an interleaved manner.
- **Chain of Thought (CoT):** A method that encourages the model to break down complex problems into intermediate steps.
- **Self-Reflection:** A technique where the model critiques its own previous outputs to improve accuracy.
- **Token Usage & Latency:** The primary constraints in designing agentic loops, requiring a balance between reasoning depth and cost.
- **Stop Sequences:** Technical markers used to signal the end of a model's thought process or action call.
