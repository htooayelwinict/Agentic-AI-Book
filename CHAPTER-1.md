# Agentic AI ဆိုတာဘာလဲ 

## Agentic နှင့် AI 

	Agentic ဆိုသည့် ဝေါဟာရနှင့် AI ကို ပေါင်းစပ်ထားခြင်းသည် ပုံမှန်အားဖြင့်မူ များစွာထူးဆန်းလှသည်မဟုတ်၊ ယနေ့လူသိများ ခေတ်စားနေသော AI Agents များကို ဆိုလိုခြင်းပင်ဖြစ်ကြောင်း အလွယ်တကူပင် သိရှိနိုင်ပေသည်။ သို့ရာတွင် ထိုအချက်ကို အနည်းငယ် ပိုမိုဂရုတစိုက် လေ့လာကြည့်သောအခါ နည်းပညာဆိုင်ရာ သဘောတရားကွဲလွဲမှုများကိုလည်း သိမြင်နိုင်ပေသည်။ 
	AI သည် ဉာဏ်ရည်အတုယောင်ဟု ရှေးကာလ နည်းပညာနယ်ပယ်၌ လက်ခံထားကြရာ၊ ယနေ့ခေတ်အခါတွင်လည်း လူတို့သည် AI ကို လူသားတစ်ဦးကဲ့သို့ပင် တွေးတောနိုင်သော နည်းပညာအဖြစ် လက်ခံလာကြပေသည်။ 
	သို့သော် ကျွန်ုပ်ယခုရေးသားသော စာအုပ်တွင်မူ AI ဆိုသည့် ဝေါဟာရကို ဉာဏ်ရည်တုဆိုသည့်အရာထက် LLM (Large Language Models) များစွာကိုသာ ရည်ညွှန်းကြောင်း ဦးစွာပြောကြားလိုပေသည်။ ထို LLM များသည် natural language processing ဟု ခေါ်ဆိုသော လူ့သဘာဝ ဘာသာစကားများကို တွက်ချက်အဖြေရှာခြင်း နည်းပညာရပ်များစွာထဲမှ တစ်ခုအစပြုရာ ၂၀၁၇ ခုနှစ်ကာလပိုင်းတွင် ပေါ်ပေါက်လာခဲ့သည့် Transformer ဆိုသည့် နည်းပညာဗိသုကာတစ်ရပ်ပင် ဖြစ်လေသည်။ ထိုနည်းပညာဗိသုကာတွင် Attention Mechanism ဟု ခေါ်သော algorithm ဖြင့် တည်ဆောက်ထားသည့် vectors linear algebra များစွာ ရှိလေသည်။ ထို algorithm ဖြင့် Cosine similarity ကို တွက်ချက်တိုင်းတာ၍ စကားလုံးများ တစ်လုံးနှင့်တစ်လုံး အနီးအဝေး ဆင်တူမှုများကို တွက်ချက်ကာ စာတစ်လုံးမှ အခြားစာတစ်လုံး၊ စာလုံးများမှ စာကြောင်းတစ်ခု ဖြစ်လာအောင် ဖော်ပြပေးသည့် လုပ်ဆောင်ချက်များစွာကို ပြုလုပ်နိုင်ပေသည်။ ယခုစာအုပ်သည် machine learning နှင့် NLP ဆိုင်ရာ သုတေသနစာတမ်း မဟုတ်သည့်အတွက် ဤမျှလောက်နှင့်ပင် LLM ဆိုသည့် သဘောတရားအနက်ကို ဖော်ပြလိုပေသည်။ 
	ထို့ကြောင့် အတိုချုပ်သော် AI သည် LLM ဖြစ်၍ LLM သည် self-Attention Mechanism ဖြင့် တည်ဆောက်ထားသည်ဟုပင် အလွယ်မှတ်ယူကြပါကုန်လော့။ 
	Agentic ဟူသော ဝေါဟာရအတွက်မူ များစွာအဓိပ္ပာယ်ဖွင့်ဆိုရန် လိုမည်မဟုတ်ပေ။ မိမိကိုယ်စား အလုပ်ကိစ္စများစွာကို ဆောင်ရွက်ပေးသည်ဟုသာ မှတ်သားရန် လွယ်ကူလှပေသည်။ 
	
	သို့ဖြစ်၍ Agentic AI သည် နည်းပညာလုပ်ငန်းဆောင်တာများကို အသုံးပြုသူကိုယ်စား LLM များကို အသုံးပြုပြီး ဆောင်ရွက်ပေးသော နည်းပညာတစ်ခုသာ ဖြစ်ပေသည်။ ဤတွင် AI workflow ဟူသော Automation နည်းပညာများနှင့် မရှုပ်ထွေးစေလိုပေ။ Agent နှင့် Workflow သည် များစွာကွဲပြားလှပေသည်။
	AI workflows များတွင်မူ လုပ်ဆောင်ချက်များသည် Dynamic routing (များစွာသော လုပ်ငန်းဆောင်တာ တာဝန်ခွဲများ)၊ conditional branching (အခြေအနေပေါ်မူတည်ပြီး ခွဲခြားတည်ဆောက်မှုများ) ပါဝင်ကြသော်လည်း အကြိမ်တရာ တထောင်မက အဖန်ဖန်အခါခါ မည်မျှပင် လုပ်ဆောင်သည်ဖြစ်စေ တူညီသော logic (deterministic) အတိုင်းသာ အလုပ်လုပ်ကြလေသည်။ သို့ရာတွင် ကိုယ်ပိုင်ဆုံးဖြတ်ချက်ချနိုင်စွမ်း (autonomy)၊ ရည်မှန်းချက်အလိုက် လုပ်ဆောင်နိုင်စွမ်း (goal-oriented reasoning) နှင့် environment (အလုပ်လုပ်နေသည့် runtime အခြေအနေများစွာ) ကို တုံ့ပြန်ပြောင်းလဲနိုင်သည့် Agent runtime မရှိပေ။ 
	တနည်းအားဖြင့်မူ LLM output များမှ ရရှိလာသော ရလဒ်ပေါ်မူတည်ပြီး တည်ဆောက်မှုအသစ်အဆန်းများ ထည့်သွင်းမထားသော ဆုံးဖြတ်ချက်များဖြင့် အဆုံးသတ်ရလဒ်ကိုမူ ပေးစွမ်းနိုင်ခြင်း မရှိကြချေ။


### `AI automation workflow framework တစ်ခုဖြစ်သည့် n8n ကို အခြေပြု၍ အောက်ပါဥပမာများကို လေ့လာနိုင်ပေသည်။ `

`1. Email Classifier များ`
   `Gmail မှ ရရှိလာသော email များကို LLM ဖြင့် အကြောင်းအရာအလိုက် ခွဲခြမ်းစိတ်ဖြာပြီး၊ အလုပ်နှင့်ဆိုင်သော email၊ customer support email၊ invoice email၊ marketing email စသဖြင့် အမျိုးအစား ခွဲခြားနိုင်ပေသည်။ ထို့နောက်တွင် သတ်မှတ်ထားသော deterministic logic အတိုင်း သက်ဆိုင်ရာ folder များဆီသို့ အလိုအလျောက် ရွှေ့ပြောင်းပေးနိုင်ပေသည်။ ဥပမာအားဖြင့် invoice ပါသော email များကို “Invoices” folder သို့လည်းကောင်း၊ customer complaint email များကို “Support” folder သို့လည်းကောင်း auto-move ပြုလုပ်နိုင်ပေသည်။ ထိုသို့ဖြင့် email များကို အသုံးပြုသူကိုယ်တိုင် လက်ဖြင့်စီရန် မလိုတော့ဘဲ အချိန်ကုန်သက်သာစေလေသည်။`

`2. Lead Enrichment`
   `CRM application ထဲသို့ customer lead အသစ်တစ်ခု ရောက်လာပါက n8n workflow မှတစ်ဆင့် external API များကို ခေါ်ယူပြီး company name၊ website၊ industry၊ company size၊ location စသည့် data အချက်အလက်များကို အလိုအလျောက် ရယူနိုင်ပေသည်။ ထို့နောက် ရရှိလာသော data များကို သတ်မှတ်ထားသော field mapping အတိုင်း CRM database ထဲတွင် ထည့်သွင်းပြီး update ပြုလုပ်နိုင်ပေသည်။ ဥပမာအားဖြင့် lead တစ်ယောက်၏ email domain ကို ကြည့်ပြီး သူနှင့် သက်ဆိုင်သော company information ကို ပြန်လည်ဖြည့်စွက်ပေးနိုင်ပေသည်။ ဤအလိုအလျောက် ပြုလုပ်နိုင်သော လုပ်ငန်းစဉ်သည် sales team အတွက် lead quality ကို မြှင့်တင်ပေးပြီး အသုံးပြုသူကိုယ်တိုင်က manual data entry လုပ်ရသော အချိန်ကို လျှော့ချပေးနိုင်လေသည်။`

`3. Content Repurposing`
  ` Blog post တစ်ခု publish လုပ်ပြီးသည်နှင့် n8n workflow သည် ထို content ကို social media platform များအတွက် သင့်လျော်သော format များအဖြစ် ပြန်လည်ပြုပြင်ပြောင်းလဲနိုင်ပါသေးသည်။ ဥပမာအားဖြင့် blog post အရှည်တစ်ခုကို Facebook post၊ LinkedIn post၊ X/Twitter short post၊ newsletter snippet စသည့် ပုံစံများအဖြစ် ပြန်လည်ရေးသားနိုင်ပေသည်။ ထို့နောက်မူ platform တစ်ခုချင်းစီအတွက် သတ်မှတ်ထားသော schedule အတိုင်း အလိုအလျောက် post တင်ရန်အတွက် အစီအစဉ် ချမှတ်နိုင်ပေသည်။ Content တစ်ခုကို platform များစွာတွင် ပြန်လည်အသုံးပြုနိုင်ပြီး marketing workflow ကို ပိုမိုစနစ်တကျ အကောင်အထည်ဖော်နိုင်ပေသည်။ `

`4. Invoice Processing`
   `Email attachment အနေဖြင့် ရောက်လာသော invoice PDF များကို n8n workflow မှ ဆွဲထုတ်ကာ extract လုပ်ပြီး invoice number၊ vendor name၊ invoice date၊ due date၊ total amount စသည့် အချက်အလက်များကို ဖတ်ယူနိုင်ပေသည်။ ထို့နောက် predefined mapping အတိုင်း accounting software ထဲသို့ အချက်အလက်များ ထည့်သွင်းခြင်း data entry ကို အလိုအလျောက် ပြုလုပ်နိုင်ပေသည်။ ဥပမာအားဖြင့် supplier တစ်ဦးမှ invoice ပို့လာပါက accounting team က manual copy-paste လုပ်စရာမလိုဘဲ invoice data ကို system ထဲသို့ LLM model ၏ စွမ်းဆောင်ရည်အတိုင်း တိကျစွာ ထည့်သွင်းနိုင်ပေသည်။ ဤလုပ်ငန်းစဉ်သည်ပင်လျှင် human error ဟုခေါ်သော လူသားအမှားကို လျှော့ချပေးပြီး finance operation ကို ပိုမိုမြန်ဆန်စေလေသည်။`

`5. Customer Support Triage`
  ` Customer support ticket များ ရောက်လာသောအခါ n8n workflow သည် keyword matching သို့မဟုတ် LLM classification ဖြင့် ticket ၏ အရေးကြီးမှုနှင့် သက်ဆိုင်ရာအဖွဲ့ကို ခွဲခြားနိုင်ပေသည်။ ဥပမာအားဖြင့် “payment failed”၊ “account locked”၊ “service down” စသည့် ticket များကို high priority အဖြစ် သတ်မှတ်ပြီး support team သို့မဟုတ် engineering team ၏ hotfix Slack channel သို့ အလိုအလျောက် ပို့ဆောင်နိုင်ပေသည်။ General inquiry များကိုမူ normal priority အဖြစ် သတ်မှတ်နိုင်ပေသည်။ Urgent issue များကို မြန်မြန်ဆန်ဆန် တုံ့ပြန်နိုင်ပြီး customer support process ကို ပိုမိုထိရောက်စေလေသည်။`

	အထက်ပါ ဥပမာများအားလုံးကို ကြည့်ပါက n8n ကဲ့သို့သော AI automation workflows များတွင် dynamic routing၊ conditional branching နှင့် LLM-based classification များ ပါဝင်နိုင်သော်လည်း ၎င်းတို့၏ အခြေခံလုပ်ဆောင်ပုံမှာ ကြိုတင်သတ်မှတ်ထားသော deterministic logic အတိုင်းသာ လည်ပတ်နေကြောင်း တွေ့မြင်နိုင်ပေသည်။ ဆိုလိုသည်မှာ workflow တစ်ခုသည် email အမျိုးအစားခွဲခြားခြင်း၊ lead data ဖြည့်စွက်ခြင်း၊ content format ပြောင်းလဲခြင်း၊ invoice data ထည့်သွင်းခြင်း သို့မဟုတ် support ticket routing ပြုလုပ်ခြင်းများကို ထပ်ခါထပ်ခါ အကြိမ်များစွာ လုပ်ဆောင်နိုင်သော်လည်း ၎င်းသည် မူလတည်ဆောက်ထားသော rule၊ condition၊ mapping နှင့် trigger များကိုသာ လိုက်နာလုပ်ဆောင်ခြင်း ဖြစ်ပေသည်။ ထို့ကြောင့် workflow သည် LLM output တစ်ခုမှ ရလဒ်ကို အသုံးချနိုင်သော်လည်း ကိုယ်ပိုင်ရည်မှန်းချက် ချမှတ်ခြင်း၊ runtime environment အပြောင်းအလဲများအပေါ် မူတည်၍ မူလလုပ်ဆောင်ပုံကို ကိုယ်တိုင် ပြန်လည်ဒီဇိုင်းဆွဲခြင်း၊ သို့မဟုတ် မထည့်သွင်းထားသော ဆုံးဖြတ်ချက်အသစ်များကို ကိုယ်ပိုင် reasoning ဖြင့် တည်ဆောက်ခြင်း မပြုလုပ်နိုင်ပေ။ ထို့ကြောင့် n8n workflows များသည် AI ကို အသုံးပြုထားသည့် automation စနစ်များ ဖြစ်နိုင်သော်လည်း အပြည့်အဝ autonomy ရှိသော AI agent များမဟုတ်ဘဲ၊ ကြိုတင်တည်ဆောက်ထားသော logic အတွင်းတွင်သာ အလုပ်လုပ်သော deterministic workflow systems များဖြစ်ကြောင်း ဆိုနိုင်ပေသည်။
